About Sphinx:

  • Безплатний
  • Open-source
  • Швидкий пошук та індексування
  • Досить добре розширюється
  • Найбільший Sphinx інстанс – boardreader.com, із 3,000,000,000 документів (Біля 4 TB тексту)
  • Найбільш зайнятий Sphinx інстанс – craigslist.org , із 50,000,000 запитів/день (560/сек)

Можливості

  • Швидке індексування
  • Швидкий пошук
  • Розподілений пошук
  • Full-text query language (field limits, phrase queries, etc)
  • Ваги(weights) для полів
  • Не текстові пошукові конструкції (WHERE authorid=123)
  • Native APIs для багатьох мов – PHP, Perl, Python, Ruby, Java..
  • Різні ранкери релевантності
  • Встроїні російський, англійський і чешський стеммери та препроцесори Soundex и Metaphone (для порівняння англійських слів по звучанню). Стеммери для інших мов теж можна підключити, тільки треба скомпілювати Sphinx із флагом --with-libstemmer.
  • Синоніми, причому є звичайні, а є «tokenizing exceptions», які можуть включати в себе спец символи.
  • Також є «blend_chars» — символи, які водночас є і розділювачими і входять в слова — наприклад, щоб «AT&T» перетворилась в слова «AT», «T» і «AT&T».
  • Уміє інфіксний пошук (для тих хто хотів швидко шукати по підстроці!), індексувати по сполученнях і навіть по вмісту заздалегідь заданих HTML-тегів. Підсвітка слів в результатах.
  • Підтримка неповнотекстових full-scan запитів
  • SQL протокол, Sphinx може фільтрувати, сортувати, групувати результат пошуку (WHERE, ORDER, GROUP)
  • Sphinx може зберігати додаткові атрибути в документах – аналог SQL стовпців

    • Дозволяє вибрати ранкери релевантності на основі різних функцій:

    • SPH_RANK_NONE
    • SPH_RANK_WORDCOUNT
    • SPH_RANK_FIELDMASK
    • SPH_RANK_PROXIMITY
    • SPH_RANK_MATCHANY
    • SPH_RANK_PROXIMITY_BM25
    • SPH_RANK_BM25
    • SPH_RANK_SPH04

    About ranking

    Multi-queries із різними оптимізаціями

    Дозволяють виконувати кілька наборів запитів у Sphinx які додатково оптимізовуються:
    • Common query optimization (added in 0.9.8) - якщо відрізняються тільки режими сортування/групування, повнотекстовий запит виконується тільки раз
    • Subtree caching – якщо запити мають схожі частини ці частини будуть закешовні

    Підтримка SELECT

    • Все що може бути зроблене через single table SELECT.
    • Може вичислювати довільні арифметичні вирази (a*b+c*log(@weight+1.234)-5)
    • Може робити WHERE, ORDER BY, GROUP BY, COUNT(*)/COUNT(DISTINCT), SUM()/AVG()
    • LIMIT Запити які в результаті ліміту обрізають частину результату кешуються і наступного разу невиконуються.
    • Дуже швидкий full scan:
    2.5 млн записів/документів
SELECT id FROM test1 ORDER BY touched DESC
– MySQL: 1.87 sec
– Sphinx: 0.87 sec
SELECT field1, COUNT(*) AS q FROM test1 GROUP
BY field1 ORDER BY q DESC
– MySQL: 1.10 sec
– Sphinx: 0.33 sec

    Недоліки:

    • Спочатку обновлюваних (realtime) індексів в Sphinx небуло, єдине що він умів — побудувати монолітний індекс і потім по ньому шукати.
    • Не вміє нормально оновлювати/видаляти документи в індексі, нормально тільки додає.
    • Realtime індекси підтримують не все — наприклад, не підтримують префікси/інфікси і MVA.
    • Невеликий бардак із інтерфейсами та фічами: оновлення реалтайм індексу тільки через SphinxQL; синтаксис пошуку в усіх трьох інтерфейсах різний (звичайний, SphinxQL, SphinxSE); 5 різних режимів пошуку функціонал яких схожий (4 obsolete)
    • indexer невміє перебудовувати реалтайм індекси
    • TRUNCATE індексу через SphinxQL зробити неможна.
